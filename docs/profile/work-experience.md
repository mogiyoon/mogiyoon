## 딥트레이드 테크놀러지스

- **기간**: 2025.09 ~ 재직 중
- **직책**: 프론트엔드 개발자

### 프로젝트: 당신의 연금 (React Native)

7건의 주요 개선 과제를 수행함.

#### 1. API 분리 및 프리페칭 도입

- **문제**: 메인 화면에서 호출하는 유저 정보 API가 2~3초 소요되어 메인 화면 전체 로딩이 그만큼 지연됨.
- **해결**: 메인 화면에 필요한 데이터를 추려 백엔드와 함께 경량 API를 신규 설계함. 무거운 기존 API는 호출 순서를 루트 렌더 시점으로 앞당겨 백그라운드에서 프리페칭되도록 변경함. 해당 API를 중복 호출하던 다른 페이지도 함께 리팩토링함.
- **결과**: 메인 화면 로딩 2~3초 → 0.6초로 개선됨. 중복 호출 페이지는 로딩이 없거나 0.3~0.4초로 단축됨.

#### 2. 테스트 코드 도입 및 강화학습 테스트 도입

- **문제**: 일부 기능은 시각적 판단이 필요하여 기존 방식으로는 자동화 테스트가 어려웠음.
- **해결**: Mock 기능 추가로 E2E 테스트 도입이 가능해져 E2E(Detox)와 Jest를 도입함. 주요 기능·컴포넌트 위주로 테스트 코드를 작성함. 시각적 판단이 필요한 테스트에는 강화학습을 적용한 CNN 모델을 도입해 테스트를 수행함.
- **결과**: 주요 기능·컴포넌트 테스트 자동화가 달성되어 오류 발생률이 감소함.

#### 3. 중복 호출 API 제거 및 Tanstack Query 캐시 관리

- **문제**: 자주 변하지 않는 값을 포커싱마다 요청하는 API, 특수한 상태에만 의존하는 API로 인해 수십 회의 중복 요청이 발생함.
- **해결**: 자주 변하지 않는 API와 특수 상태 의존 API를 파악함. Tanstack Query의 stale time 설정으로 포커싱마다 재호출되던 불필요한 요청을 차단함. mutate 이후 query client로 관련 데이터를 stale 처리하여 필요한 시점에만 갱신되도록 변경함. useEffect로 복잡하게 관리되던 코드를 제거함.
- **결과**: API 호출 횟수 수십 회 → 3~4회로 단축됨. 상태 관리 생산성 향상.

#### 4. Mock 테스트 기능 추가

- **문제**: 자주 변하는 프론트엔드 UI 상태를 임의로 확인하는 기능이 부재했고, 테스트용 계좌의 기능적 한계로 다양한 시나리오 테스트가 어려웠음.
- **해결**: API Mock Adaptor를 도입하고 내부 주요 상태를 원하는 값으로 전환하는 기능을 구현함. API 로그 기준 사용률 80%에 해당하는 API에 대한 mock을 추가함.
- **결과**: 기존 개발 방식 대비 개발 생산성 증가 및 오류 발생률 저하.

#### 5. 번들 최적화

- **문제**: 테스트 코드 추가로 production 빌드 번들에 테스트 관련 로직이 포함되어 번들 크기가 증가함.
- **해결**: babel.config, metro.config를 활용해 빌드 시 테스트 관련 소스 파일을 stub 파일로 자동 치환함. console 관련 코드 및 배포에 불필요한 코드를 제거함. atlas, hbc map으로 번들 결과를 분석하며 최적화 효과를 검증함.
- **결과**: 번들 용량 110MB → 80MB로 감소함.

#### 6. 레거시 코드 수정 및 보완

- **문제**: 구글 플레이 스토어 16kb 이슈, 레거시 라이브러리 충돌, 함수·컴포넌트 중복으로 인해 개발 시간이 증가함.
- **해결**: Expo 52 → 54 마이그레이션으로 16kb 이슈를 해결하고 미지원 라이브러리를 삭제하며 코드를 수정함. 주요 파일의 소스 코드를 공통 함수·훅·컴포넌트로 분리하여 리팩토링함.
- **결과**: 페이지 파일 소스 코드 분량 1/4로 감소. 유지보수성·가독성 향상.

#### 7. 배포 파이프라인 구축

- **문제**: 테스트 자동화를 배포 과정에 추가하고 실시간 OTA 배포가 필요해졌으나 관련 파이프라인이 없었음.
- **해결**: npm 명령어를 통한 자동화 테스트를 파이프라인에 추가하고 EAS 빌드·배포 기능을 구성함. OTA 업데이트를 위한 EAS 채널을 추가하고 테스트용 앱을 내부 배포함. 내부 테스트는 babel.config·metro.config를 활용해 번들에 mock 기능이 포함되도록 설정함. 로컬 → 내부 테스트 → 최종 테스트 → 배포의 4단계 파이프라인을 완성함.
- **결과**: TypeScript 코드 관련 장애·버그 대처 시간이 2일(iOS 앱 심사 기준) → 1시간으로 감소함.
